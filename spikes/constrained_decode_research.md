# Constrained-Decode Logit Masking for Smelt

Build-vs-adopt research, April 2026. Decode budget: ~5 ms/token (~200 tok/s) for Qwen 3.5 2B on Apple Silicon. Vocab ranges of interest: Llama 3.x ≈ 128k, Qwen 3.5 ≈ 152k. Target downstream contract: Pi's `AssistantMessageEvent` stream (`toolcall_start → toolcall_delta* → toolcall_end → done`).

---

## 1. Landscape & Mechanics

### xgrammar (CMU Catalyst / MLC, Dong et al. 2024)

xgrammar compiles a CFG (derived from the JSON Schema) into a **byte-level pushdown automaton (PDA)** and walks it over generated bytes. Vocabulary is partitioned into **context-independent** tokens (validity decided by PDA stack-top alone) and **context-dependent** tokens (need the full stack). For JSON-on-Llama-3.1 they report **~1,134 of 128k tokens are context-dependent**, and "context expansion" shrinks even that to ~120 (~0.1%). The remaining ≥99% live in an **adaptive token mask cache** indexed by stack-top, stored per state as rejection-list, acceptance-list, or full bitset — whichever is smallest (cited compression: **160 MB → 0.46 MB**). Per-token mask cost is <40 µs and end-to-end TPOT is essentially unchanged: 6.3 ms vs 6.2 ms unconstrained on Llama-3.1-8B, batch 1, H100. They claim up to 100× speedup on grammar walks vs prior systems and up to 80× end-to-end vs older guided backends.[^xg-paper][^xg-blog][^xg-repo]

xgrammar is now the **default structured-output backend in vLLM (Dec 2024), SGLang (Nov 2024), TensorRT-LLM (Jan 2025), and MLC-LLM (Nov 2024)**.[^xg-repo] Apache-2.0, ~50% C++ with Python/JS bindings, CMake. No documented stable C ABI (Python extension via pybind). Community Rust bindings (`xgrammar-rs`) exist; a `[metal]` extra targets MPS; iOS is not advertised.[^xg-repo]

### llguidance (Microsoft, Mosko et al.)

llguidance is the engine behind Microsoft's `guidance` library, head-to-head with xgrammar. It uses a **two-stage lexer + Earley parser**: derivative-of-regex DFA lexer handles 99–99.9% of token checks; the Earley parser engages only at ambiguous boundaries ("0.1–1% of token checks"). Vocab is a **byte-trie**, traversed at ~13 cycles/node, entire sub-trees pruned the moment a byte fails. **No precomputed mask cache** — masks built on-the-fly each step. Cited mean grammar startup ~2 ms vs "1000×" longer for Outlines.[^llg-blog][^llg-repo]

Headline: **~50 µs single-core CPU per mask at 128k vocab** (MaskBench JSON-schema), <1% of masks >1 ms, 0.001% >10 ms (capped <30 ms). Claims other libraries are **10–1000× slower**.[^llg-blog] Rust (~86% of repo). A first-class **C header (`parser/llguidance.h`)** with `c_sample.cpp` is in-tree — cleanest C ABI in the field.[^llg-repo] Supports a large JSON-Schema subset, regex, and a Lark-like CFG with embedded JSON-Schema/regex.

### outlines (.txt / dottxt-ai, Willard & Louf 2023)

Outlines popularized the **regex-FSM-over-vocabulary** approach (Willard & Louf, arXiv:2307.09702[^willard-louf]): convert schema → regex → FSM, and for each FSM state precompute the set of valid vocab tokens. Decode-time masking is a per-state hash-table lookup — extremely cheap **once built**. Cost shifts to compilation. **GitHub issue #658** documents a Pydantic schema with `conlist(constr(max_length=100), max_length=5)` that hung ~1600 s on 32 GB and OOM'd; shrinking `max_length` reduced time/memory **exponentially**.[^outlines-658] Structural problem: bounded-length string constraints multiplied with list bounds blow up the FSM. JSONSchemaBench measured Outlines at **3.5–8 s compilation** and 30–46 ms TPOT vs Guidance's 6.4–9.5 ms.[^jsb-paper] Functionally superseded for low-latency serving by xgrammar/llguidance.

### lm-format-enforcer (Noam Gat)

Older. **Character-level parser** (one instance per active schema branch) advanced character-by-character, intersected each step with a **prebuilt tokenizer prefix tree** by recursive co-traversal.[^lmfe-repo] The recursion enumerates every token whose byte sequence is a path through both trees. Per-character state plus whitespace/optional-field flexibility makes it tolerant of "model inserted a space" cases that strict FSMs reject — the main reason it survives in HuggingFace/langchain integrations despite far lower throughput. MaskBench places it in the "10–1000× slower than llguidance" bucket.[^llg-blog]

### OpenAI Structured Outputs (Aug 2024) and Anthropic Strict Tool Use

OpenAI's launch post[^openai-so] confirms a CFG-based engine (explicitly contrasted with Outlines-style FSMs because CFGs cover schema features regexes cannot). The implementation pre-compiles each unique schema, **caches the result**, and then masks each step using the cached structure. **First-request latency is the visible cost**: typical schemas <10 s, complex ones up to ~60 s.[^openai-cfg-thread] Cached calls show no measurable extra TPOT.

Anthropic's "strict tool use" (`strict: true` on a tool definition) is documented as **grammar-constrained sampling** sharing the structured-outputs pipeline, with a 24-hour grammar cache.[^anthropic-strict] Neither vendor has published per-token µs or the engine internals.

## 2. Performance Reality

### Per-token mask cost (the headline number)

| System | Per-token mask cost @ ~128k vocab | Source |
|---|---|---|
| xgrammar (JSON) | <40 µs (claimed); TPOT delta <0.1 ms on Llama-3.1-8B H100 | [^xg-paper] |
| llguidance | ~50 µs single-core mean (MaskBench); 99th-pct <1 ms; 99.999th <10 ms | [^llg-blog] |
| Outlines (post-build) | sub-ms hash-lookup, but **3.5–8 s build** per schema | [^jsb-paper] |
| llama.cpp grammars | ~10–30 ms per token at 128k (MaskBench, "10–1000× slower than llguidance") | [^llg-blog] |
| lm-format-enforcer | not published; same "10–1000× slower" bucket | [^llg-blog] |

Both leaders sit in the 30–60 µs/token range at 128k vocab; **152k vocab (Qwen 3.5) is ~19% larger** and the dominant trie-walk / bitmask-fill costs are roughly linear, so expect ~60–70 µs/token. Neither has been published explicitly at 152k.

### Cold-start (grammar compilation)

- **llguidance**: ~2 ms mean.[^llg-blog]
- **xgrammar**: not published as a single number, but their adaptive-cache build is reported as a heavier upfront cost — llguidance's blog calls it "seconds, sometimes minutes" on pathological grammars.[^llg-blog]
- **Outlines**: 3.5–8 s on JSONSchemaBench, multi-minute on issue #658-class schemas.[^jsb-paper][^outlines-658]
- **OpenAI Structured Outputs**: <10 s typical, up to ~60 s for complex schemas.[^openai-cfg-thread]

### Worst-case schema breakers

- **Recursive schemas**: every FSM-style system rejects, unrolls to a bound, or falls back to PDA/stack. xgrammar handles natively (PDA); llguidance via Earley; Outlines needs bounding.
- **Large enums** (>1k strings): xgrammar's adaptive cache compresses well; Outlines' regex union grows linearly in states; llguidance lexes the union as one regex derivative.
- **Bounded-length strings × bounded list** (`conlist(constr(max_length=100), max_length=5)`): the canonical FSM-state-explosion case (Outlines #658).[^outlines-658] xgrammar/llguidance handle bounded counts at PDA/parser level instead of baking them into a regex.
- **Deep nesting / `anyOf`/`oneOf` with many branches**: PDA and Earley systems pay growth proportional to live branches — this is the source of llguidance's 99th-percentile (1 ms) and 99.999th-percentile (10 ms) tail spikes.[^llg-blog]
- **Unicode-heavy `format`** (`email`, `date-time`): JSONSchemaBench found best-case **~38–41% empirical schema coverage** on the GitHub-Hard split across all six tested frameworks. Constrained decoding is **not** schema-complete even where per-token speed is fine.[^jsb-paper]

### Where CPU time goes within a step

llguidance: **>99% of token validity checks resolve at the lexer** (regex-derivative DFA over a trie node), Earley parser handles 0.1–1%, bitmask fill is the trailing cost. xgrammar: cache lookup (hash + bitset OR) dominates for ≥99% of tokens; the few hundred context-dependent tokens walk the PDA each step. In both, bitmask write (152k bits = 19 kB) is not the bottleneck.

### Throughput regression and the 5 ms budget

xgrammar reports **TPOT 6.3 ms vs 6.2 ms unconstrained** at batch 1 on H100, Llama-3.1-8B (~1.6%).[^xg-paper] llguidance can support batch sizes up to **~3,200 in parallel** at 16 cores without slowing GPU inference.[^llg-blog] SGLang's **jump-forward decoding** (skip tokens when only one continuation is grammar-valid) reports up to **2× lower latency, 2.5× higher throughput** on JSON outputs vs unconstrained.[^sglang-fsm]

At our 5 ms/token budget, mask compute at 50–70 µs is **~1.0–1.4%** of the decode step. With unified memory the logits are CPU-readable at command-buffer completion, and the *next* mask (depends only on the token just sampled, not on its logits) builds on a separate CPU thread overlapped with the GPU launch of step `t+1`. Mask never extends the critical path — except in (a) speculative decoding where draft tokens multiply cost, (b) batch >>1 with tail-latency schemas, or (c) the rare 99.999th-percentile schema-state spike at 10–30 ms. (c) is the only realistic budget-buster.

## 3. Failure Modes

### Tokenizer-boundary vs grammar-boundary mismatch

Classic case: BPE tokenizes `"foo","bar"` into a single `","` token that spans the closing-quote of one JSON value and the comma starting the next array element. A token-level FSM (Outlines circa 2023) would need a single FSM state that simultaneously "closes a string" and "opens an array element" — different non-terminals.

xgrammar walks the PDA at the **byte** level (one byte per PDA step, many bytes per token). Each candidate token's byte sequence is hypothetically advanced through the PDA; the adaptive-cache key is the PDA stack-top, so a token bridging two stack-top states is correctly classified context-dependent. llguidance walks **bytes through the lexer DFA** and rolls back via the Earley parser; trie iteration means a multi-byte BPE token is naturally a trie path crossing logical char boundaries. lm-format-enforcer's prebuilt trie was designed exactly for this.[^xg-paper][^llg-blog][^lmfe-repo]

### Byte-fallback tokens in Qwen 3.x and Llama 3.x

Both tokenizers are byte-level BPE and emit raw byte tokens when sequences don't merge into vocab entries. Qwen's docs warn[^qwen-tok] that mid-stream decode may produce U+FFFD because a multi-byte UTF-8 character splits across tokens. The practical issue: **a byte-fallback token may be an invalid-UTF-8 prefix that's only valid once the next byte arrives**. Byte-level engines (xgrammar, llguidance) handle this naturally — the grammar walk is on bytes, and a half-character token just leaves the lexer in an intermediate UTF-8 state. Character-level engines (lm-format-enforcer, older Outlines configs) are fragile here.[^byte-bpe-paper]

### Whitespace ambiguity in JSON

Strict FSM systems either pin a whitespace pattern (forcing emission) or admit `\s*` everywhere (branching cost). lm-format-enforcer deliberately lets the model control whitespace/optionality/field-ordering[^lmfe-repo]; xgrammar and llguidance default to permissive whitespace, with stricter modes available.

### "Valid garbage" — does masking degrade quality?

Tam et al. ("Let Me Speak Freely?", EMNLP 2024 Industry)[^tam-paper] reported large drops on reasoning under JSON-mode: GPT-3.5 GSM8K 76.6% → 49.3%, Claude-3-Haiku GSM8K 86.5% → 23.4%, Llama-3-8B GSM8K 74.7% → 48.9%, Last Letter 70.1% → 28.0%. Interpretation: JSON-mode forces "answer-first" output, killing chain-of-thought.

The dottxt rebuttal "Say What You Mean"[^dottxt-rebuttal] re-ran the Llama-3-8B experiments and found (a) the original used **fundamentally different prompts** for structured vs unstructured (the structured prompt didn't even mention JSON or the schema), and (b) Claude-as-parser was used to rescue malformed unstructured answers, flattering the unstructured baseline. With matched prompts and a fair regex parser: GSM8K 0.77 → 0.78, Last Letter 0.73 → 0.77, Shuffle Object 0.41 → 0.44 — structured **slightly better on every task**.

April 2026 synthesis: at modern model sizes, schema constraint per se does not measurably degrade reasoning *if* the schema permits chain-of-thought (e.g. a `reasoning` string field before `answer`). The "Let Me Speak Freely" effect is a prompting/schema-design artifact, not a property of constrained decoding. JSONSchemaBench[^jsb-paper] independently shows the harder problem is *schema coverage*, not quality regression.

### Streaming: when is it safe to emit a `toolcall_delta`?

Three boundary classes: (1) **token** boundaries — never rewinds, but a `","` token spans two JSON tokens; (2) **lexer-stable** boundaries — llguidance's lexer hits an "end of token" state at every JSON structural token (`{`, `,`, `:`, `"`-close, `}`), the natural safe-emit point; (3) **grammar-stable value-complete** events. xgrammar exposes `IsCompleted` (recently surfaced through bindings[^xg-repo]) for whole-grammar termination, not per-value. Neither engine exposes a first-class "key complete" event; you build it on top by tracking PDA/Earley transitions into JSON value-end non-terminals. **Smelt needs its own thin state-machine layer regardless of engine choice.**

## 4. Pipeline Placement on Apple Silicon

### CPU mask vs Metal-kernel mask

Qwen 3.5 logits per step: **152k × 4 B = 608 kB**, fits in half of an M3 P-core L2. Mask: **152k bits = 19 kB**, trivial. A fused Metal `mask + softmax + sample` kernel would save a round-trip, but on unified memory "GPU → CPU" is a cache-coherency event, not a copy, and at our budget the round-trip is single-digit µs. **Do mask on CPU.** Exception: if we go batched (spec-decode draft validation, parallel multi-tool generation), per-stream CPU mask scales linearly while a Metal-fused mask wouldn't — revisit at batch ≥8.

### Pipelining

Dependency chain: sample at step `t` → advance grammar by sampled token → build mask for `t+1`. Grammar-advance and mask-build **do not depend on `t+1` logits**, so they run on a CPU thread overlapped with the GPU launch of `t+1`'s matmul. Critical-path budget:

- GPU step (matmul + attn): ~4.5 ms target
- Grammar advance + mask build: 50–70 µs
- Mask apply (NEON `vbsl` over 608 kB logits + 19 kB mask): ~12 µs at ~50 GB/s sustained
- Sample (top-p over 152k post-mask): 50–100 µs

Serial sample-then-launch tail is ~150 µs. Overlap mask-build with the next GPU launch and it hides entirely. Inside-Metal masking is only worth it once we already have a fused Metal sampler — at that point AND'ing 19 kB into a kernel that reads 608 kB anyway is free. For v1, CPU.

## 5. Integration Paths for Smelt

### (a) Wrap xgrammar via C/C++

C++ with pybind glue; no first-class documented C ABI. Plan: build static lib, write a **thin C shim** exposing `xgrammar_compiler_compile`, `xgrammar_matcher_create/accept_token/get_mask/is_completed`. CMake → `.a`, Swift via `module.modulemap`. Apple Silicon supported (`[metal]` extra); iOS not advertised but the C++ has no Linux-isms (`CMAKE_OSX_ARCHITECTURES=arm64` should work). Hidden cost: a **tokenizer adapter** that emits xgrammar's `TokenizerInfo` for Smelt's vocab. **Estimate: 2–3 engineer-weeks** including tests and adapter.

### (b) Wrap llguidance via C ABI

Ships a stable C header (`parser/llguidance.h`) with `c_sample.cpp` — Microsoft uses it for llama.cpp integration. Build: `cargo build --release --features=cffi` produces `libllguidance.a` for `aarch64-apple-darwin` (and `aarch64-apple-ios` with `--target`). Swift integration: `module.modulemap` over the header. Same tokenizer-adapter cost. **Estimate: 1.5–2 engineer-weeks.** Less than xgrammar because the C ABI is first-class and the build is `cargo` rather than CMake.

### (c) Build a Smelt-native minimal-JSON-Schema enforcer

Auditing real tool schemas (OpenAI tools, Anthropic tools, MCP servers, ~30 schemas across fs/shell/search/git/web-fetch agents) yields the **minimum viable subset**:

- `type`: `string | number | integer | boolean | object | array | null`
- `properties`, `required`, `additionalProperties: false`
- `enum` (mostly strings, small)
- `items` (single-schema arrays)
- `minimum`/`maximum`, `minLength`/`maxLength` (rare in tool schemas)
- `oneOf`/`anyOf` at top level (tool dispatch); occasional argument unions
- `description` (prompt-only)

Not needed in practice: `pattern`, `format`, `$ref` recursion (tool schemas are flat — one nesting level max), `if`/`then`/`else`, `dependentSchemas`, `patternProperties`, `multipleOf`.

Implementation splits in two:

1. **Schema-to-PDA**: the easier half. JSON itself is ~30 productions; layering schema constraints adds a few hundred lines.
2. **Token-aligned masking**: the hard half. Needs (i) a tokenizer-byte-trie built once from Smelt's vocab, (ii) a per-step routine walking the PDA over candidate token byte sequences, (iii) caching to avoid re-walking. This is where xgrammar/llguidance derive their performance edge through years of optimization (adaptive cache layouts, lexer/parser split, slice preprocessing).

Estimate for correct-but-not-SOTA (subset above, ~200 µs/token, no jump-forward, no spec-decode rollback): **5–8 engineer-weeks** for v1 plus ongoing schema-coverage maintenance. Our 5 ms budget tolerates 200 µs (4%) comfortably — the perf gap doesn't break us.

**>70% of the engineering is tokenizer-aware masking, not the grammar engine itself.** That cost is the same regardless of grammar source.

## 6. Pi Integration Notes

### Safe emission boundaries

Pi emits partial JSON fragments. Robust pattern: **emit at lexer-stable boundaries**, not at token boundaries. After every grammar transition that closes a JSON value (`}`, `]`, `"`-string-close, number-end, `true`/`false`/`null`-end, or object-property `,`), flush bytes generated since the last emission as one delta. Each delta is then a contiguous JSON-fragment-prefix of the eventual tool call. Internal buffer is bounded by max value length (typically <1 kB).

`IsCompleted` (xgrammar) and `is_accepting` (llguidance) check whole-grammar termination, not value boundaries. We have to surface value-boundary events ourselves by inspecting the non-terminal we just exited; both engines expose enough state for it. Wrap in a Smelt-side `JSONStreamCursor` yielding `{type: "value_complete", path: "/tool/args/foo"}`.

### Multi-tool union grammar and tool-name dispatch

Natural construction: `anyOf: [tool1, tool2, ...]`. Both engines handle top-level `anyOf` natively; cost scales with live branches. The win is **early pruning** during tool-name decoding. In the OpenAI/Anthropic convention `{"name": "fs_read", "arguments": {...}}`, once `"name": "fs_read"` decodes, all other branches are inconsistent (their `name` enum literals don't match). xgrammar's PDA prunes them from the live stack; llguidance's Earley parser collapses the Earley set. **Both engines automatically narrow** to the dispatched tool's schema with zero extra work. A Smelt-native enforcer replicates this by maintaining a "live tool set" bitset and dropping tools whose name-enum literal mismatches the latest accepted byte.

### Pi event state machine

```
state=Idle
on first non-whitespace JSON byte:    emit toolcall_start
on safe-boundary event from cursor:   emit toolcall_delta(buffered_bytes)
on grammar IsCompleted=true:          emit toolcall_end
on EOS while grammar accepting:       emit done
on grammar mask-empty (no valid tok): hard error — abort, alarm
```

The mask-empty case is the operational failure mode and the metric to alert on.

## 7. Honest Verdict

### Tier-1 differentiator or table stakes?

Table stakes, with a caveat. By April 2026 every serious LLM-serving stack ships constrained JSON: OpenAI, Anthropic, Google, Mistral, vLLM, SGLang, TensorRT-LLM, MLC-LLM, llama.cpp. Apple Silicon local inference has had Outlines-via-MLX for over a year. The differentiator is not "we have constrained decoding." It's **"Pi tool registration → guaranteed-valid tool call adds <5 ms latency, never breaks on byte-fallback tokens, never hangs at compile time on user schemas."** That's a UX/operability bar — favors adopting a battle-tested engine over building.

### Smallest implementation that captures most of the value

v1:
1. Adopt **llguidance** via its C ABI.
2. Smelt-side JSON-stream cursor over llguidance parser state for `toolcall_delta` events.
3. `union-of-tools` grammar at registration; rely on llguidance's natural branch pruning for tool dispatch.
4. CPU-side mask apply, NEON-vectorized, off the GPU critical path.
5. Skip speculative decoding, jump-forward, Metal-fused sampling — v2 work.

≥95% of the value at ≤2 weeks integration cost.

### Most likely thing to go wrong six months in

**Tail-latency spikes from a user-supplied schema with high-branching `anyOf` plus bounded-length strings.** llguidance's 99.999th-percentile is 10–30 ms vs 50 µs median. A single 30 ms mask-build event blows our 5 ms budget 6× and at 200 tok/s the user sees a visible stall. The triggering schema isn't human-pathological — it's a real tool with several optional string fields with `maxLength` and an `enum`-of-many-options. Fix is reactive (per-schema timeout + relaxed-grammar fallback) and the bug report lands well before the fix. **Build the timeout-and-fallback path before v1 ships.**

Lower-probability runner-up: **byte-fallback token misalignment when Qwen splits a 3-byte UTF-8 char across two tokens.** Engines handle this if fed byte sequences correctly, but the integration plumbing — Smelt's tokenizer telling llguidance "token 12345 is bytes [0xE2, 0x82]" — is exactly where off-by-one bugs hide. Fuzz Qwen's full vocab at integration time.

### Recommendation: adopt llguidance for v1

- Cleanest C ABI in the field (1.5–2 weeks vs 5–8 for native).
- Enormous performance headroom on our budget (50 µs in a 5 ms window).
- Active Microsoft maintenance; default backend in `guidance` and growing.
- Best-in-class schema coverage (JSONSchemaBench). When a user tool schema has weird `format` or `pattern`, llguidance is more likely to handle it than a Smelt-native enforcer.
- The Rust+C-ABI integration pattern pays forward for future Rust deps.

**v2** (3–6 months out): if telemetry shows real-user schemas in the tail-latency danger zone, evaluate xgrammar's adaptive-cache approach at our schemas (its cache-hit path is ~40 µs deterministic). Second adoption is a 2-week integration, not a rewrite.

**Do not build** unless: (a) we need a deployment target where neither Rust nor xgrammar C++ can link (theoretical today), (b) a real perf regression neither engine fixes upstream, or (c) a tokenizer-aware feature neither exposes (e.g. tight spec-decode integration). None is true today.
